Hito 4: Conectar GitHub Actions con tu AWS Local (Floci)

Ahora viene el verdadero reto de arquitectura: ¿Cómo hacemos para que el "robot" de GitHub Actions (que vive en los servidores de GitHub en internet) se conecte a un contenedor de Docker (Floci) que está corriendo dentro de tu Mac local?

En el mundo real, un servidor de producción tiene una IP pública en internet y GitHub se conecta directo. Pero como tu Mac está protegida detrás del router de tu casa u oficina, GitHub Actions no puede "ver" tu localhost:4566.

La Solución Senior: Ngrok (Túnel Seguro)

Para resolver esto sin gastar en servidores reales, vamos a usar una herramienta estándar de la industria llamada Ngrok. Ngrok creará un túnel seguro y temporal que expondrá el puerto 4566 de tu Mac a internet a través de una URL pública (por ejemplo: https://un-id-aleatorio.ngrok-free.app).

De esta forma, le daremos esa URL a GitHub Actions para que pueda deployar directo en tu computadora.

Paso 2: Crear una cuenta gratuita y obtener tu AuthToken

Para poder usar Ngrok, necesitas autenticar tu terminal (política de seguridad contra spam):

    Entra a ngrok.com y regístrate con tu cuenta de GitHub (es gratis y toma un clic).

    En tu panel de Ngrok, busca la sección "Your Authtoken".

    Copia el token largo que te dan y ejecútalo en tu terminal de Mac de esta manera:


ngrok config add-authtoken TU_TOKEN_AQUÍ

ngrok http 4566

# Paso 2: Crear el archivo de Workflow en tu proyecto

GitHub Actions busca automáticamente los archivos de configuración en una ruta muy específica de tu proyecto.

    En la raíz de tu proyecto de React, crea una carpeta llamada .github.

    Dentro de esa carpeta, crea otra llamada workflows.

    Dentro de workflows, crea un archivo llamado deploy.yml.

La estructura de carpetas en tu editor debe verse exactamente así:
tu-proyecto/
└── .github/
    └── workflows/
        └── deploy.yml

Paso 3: Código del Pipeline de CI/CD

Pega el siguiente contenido dentro de tu archivo deploy.yml. Este script le dice al robot de GitHub exactamente qué hacer cada vez que subas código a la rama main:


add deploy.yml


git add .
git commit
git push origin master


# Forma 1: Revisa la terminal con el AWS CLI (La prueba rápida)

# Pregúntale directamente a tu Floci local qué archivos tiene guardados en el bucket. Ejecuta este comando en tu terminal de Mac:


aws --endpoint-url=http://localhost:4566 s3 ls s3://mi-bucket-frontend --recursive

# Si la respuesta está vacía: El pipeline falló silenciosamente o subió los archivos a una ruta incorrecta (por ejemplo, una carpeta vacía como dist/ si la compilación generó otra carpeta como build/).

# Si ves archivos (como index.html): El problema es meramente del navegador y del hosting. Pasemos a la Validación 2.

Validación 2: ¿Activaste el modo sitio web después del push?

# Si la Validación 1 te mostró archivos, ejecuta este comando obligatoriamente para decirle a Floci que se comporte como un servidor web:

aws --endpoint-url=http://localhost:4566 s3 website s3://mi-bucket-frontend --index-document index.html



# Validación 3: Probar la URL alternativa (Evitando problemas de DNS)

# Muchos navegadores (especialmente las versiones recientes de Chrome y Brave) bloquean los subdominios de localhost (bucket.localhost) por motivos de seguridad.

# Si la URL http://mi-bucket-frontend.localhost:4566/ te da un error de "No se puede acceder a este sitio", Floci ofrece una estructura de URL alternativa que no requiere configuración de DNS:

# Intenta ingresar a esta URL exacta en tu navegador:
# 👉 http://localhost:4566/mi-bucket-frontend/index.html
